	Postgres Index Performance Flashcards
	Core Concepts
	Q: What's the fundamental difference between Index Scan and Bitmap Scan?
	A: Index Scan fetches rows immediately one-by-one (streaming). Bitmap Scan builds a complete list of pages first, then fetches them all in sorted order (batching). This can be bad if limit is present



	Q: What operations are considered "batching" (non-streaming)?
	A:
		Bitmap Heap Scan - must build complete bitmap first
		BitmapOr / BitmapAnd - must scan all indexes and merge
		Sort - must see all rows to sort them (unless using index order)
		Aggregate (GROUP BY, COUNT, SUM) - must process all groups
		Hash Join - must build complete hash table from inner side
		Hash Aggregate - must build complete hash table of groups
		Materialize - buffers complete result set
		WindowAgg (without partition pruning) - may need full result

	Q: Why does batching kill LIMIT optimization?
	A: LIMIT wants to stop early. Batching strategies must complete their full work before returning ANY rows, so LIMIT can't stop them early.

	Index Scan vs Bitmap Scan
	Q: When does Postgres choose Bitmap Scan over Index Scan?
	A: When expecting MANY rows spread across MANY heap pages. Visiting pages in sorted order once is cheaper than random I/O visiting the same pages multiple times.
	Q: Why is Index Scan better for small result sets?
	A: Few rows = few random I/O operations. The overhead of building a bitmap and sorting isn't worth it. Plus it can stop early with LIMIT.
	Q: Why is Bitmap Heap Scan LIMIT-unfriendly?
	A: It must scan the entire index, build the complete bitmap, sort all page locations, and read all identified pages. LIMIT cannot stop this process early.

	OR Conditions (The Problem Child)
	Q: Why are OR conditions problematic for performance?
	A: Each OR branch typically requires a separate index scan. Results must be merged (BitmapOr), and you can't stop until ALL branches are complete. Wide ORs = many index scans = slow.
	Q: What makes an OR condition "acceptable"?
	A: ALL branches must be highly selective (return few rows). If any branch returns many rows, the entire operation becomes expensive.
	Q: How does BitmapOr work?
	A: Scans multiple indexes (one per OR branch), builds a bitmap for each, then merges all bitmaps by combining TIDs (tuple identifiers). Must complete ALL index scans before proceeding.
	Q: What happens if one OR branch is poorly selective?
	A: That branch's bitmap explodes in size, consuming memory and requiring many heap page fetches. The entire query pays the cost of the worst branch.
	Q: Can LIMIT optimize a BitmapOr query?
	A: No. BitmapOr must complete all index scans and merge all bitmaps before ANY rows are returned. LIMIT happens after all this work is done.

	AND Conditions (The Good Guys)
	Q: Why are AND conditions better than OR for performance?
	A: ANDs are selective (they narrow results). Postgres can use a single composite index OR use multiple indexes and AND their bitmaps (smaller result set). Each condition filters OUT rows.
	Q: How does a composite index help AND conditions?
	A: A composite index on (col1, col2) can satisfy WHERE col1 = x AND col2 = y with a single index scan. No bitmap merging needed.
	Q: Can AND conditions also use BitmapAnd?
	A: Yes, but it's better than BitmapOr because: (1) ANDs reduce the result set, (2) the final bitmap is smaller, (3) fewer heap pages to fetch.

	Lossy Bitmaps
	Q: What is a lossy bitmap?
	A: When work_mem is exceeded, Postgres switches from tracking individual row TIDs to tracking entire pages. Instead of "rows 1,5,7 on page 10", it becomes "page 10 (somewhere)".
	Q: Why are lossy bitmaps slow?
	A: Every row on the marked pages must be rechecked against the WHERE clause. You fetch entire pages and scan them row-by-row, even if only one row matches.
	Q: When do bitmaps become lossy?
	A: When the bitmap size exceeds work_mem. Wide ORs, poorly selective conditions, or many matching rows cause this.

	Materialize Node
	Q: When is Materialize helpful?
	A: When a SMALL result set is reused many times (like in a nested loop join). The cost of computing once and caching pays off.
	Q: When does Materialize hurt performance?
	A: When a LARGE result is materialized. It buffers everything in memory/disk before proceeding. LIMIT cannot stop early, and you pay memory/I/O costs for data you may not need.
	Q: Why is Materialize LIMIT-unfriendly?
	A: It computes and stores the ENTIRE result set before returning anything. LIMIT can only apply after materialization is complete.

	LIMIT Optimization Rules
	Q: What query patterns keep LIMIT working efficiently?
	A: Streaming execution plans: simple Index Scans, no bitmap operations, no materialize nodes, avoid OR conditions, keep subplans small and selective.
	Q: What execution nodes break LIMIT optimization?
	A: Any "must complete before continuing" node: Bitmap Heap Scan, BitmapOr, BitmapAnd, Materialize (large), Aggregate, Sort (without index), Hash Join build phase.
	Q: How can you diagnose why LIMIT isn't helping?
	A: Run EXPLAIN ANALYZE. Look for: (1) Bitmap operations, (2) Materialize nodes, (3) "actual rows" = total rows despite LIMIT, (4) sort/aggregate nodes that must see all data.


	Quick Reference
	Streaming (LIMIT-friendly):

	Index Scan
	Index Only Scan


	Seq Scan (streams, but slow)

	Batching (LIMIT-unfriendly):

	Bitmap Heap Scan
	BitmapOr / BitmapAnd
	Sort (without index order)
	Hash Aggregate
	Aggregate (most cases)
	Materialize
	WindowAgg (some cases)

	Key Insight: LIMIT optimization requires the ability to stop early. Any execution node that must "see all data before proceeding" kills this optimization.

  Q: What is the primary purpose of the InstanceService class?
  A: It's the main entry point for processing DICOM files - extracts structured data from DICOM files, validates required identifiers, creates/retrieves
  database objects, and handles integrity errors with retry logic.

  Flash Card 2
  Q: What does get_instance_file_data return and how is it used?
  A: Returns a dictionary with instance data and file type ("DCM"). Used by create_update_instance_file() to extract metadata needed for creating
  InstanceFile records in the database.

  Flash Card 3
  Q: What are ECG signals in this medical imaging system?
  A: Electrocardiogram data - electrical activity recordings of the heart stored in DICOM files and processed into structured digital signal data for
  storage and analysis.

  Flash Card 4
  Q: How does the system process ECG DICOM files?
  A: Uses ECG handler library to extract multiple channels (leads) from DICOM, converts analog waveforms to digital signal lists, and stores them as
  SeriesEcg and SeriesEcgSignal records in the database.

  Flash Card 5
  Q: What triggers ECG signal generation and what does it enable?
  A: Triggered when DICOM modality is "ECG" via task_generate_ecg_signals background task. Enables displaying ECG waveforms, heart rhythm analysis, and
  maintaining full diagnostic information from ECG machines.
